## 一、产品名称与愿景

### 1.1 产品全称

**工业逻辑竞技场**（Industrial Logic Arena，简称 **ILA**）

### 1.2 核心愿景

> 打造一款**混合现实（MR）、回合制 PvP 工业排故沙盘**，专为职业教育场景设计——让学生从被动记忆工艺流程，转变为在激烈的对抗博弈中主动构建、推理、验证**复杂系统因果逻辑**。

### 1.3 一句话定位

**"工业排故界的《炉石传说》——用有限的卡牌预算，构造一个让对手无法轻易破解的隐性故障，再亲眼看他们被逼到绝境或绝境逢生。"**

### 1.4 产品理念支柱

| 支柱 | 描述 |
|------|------|
| 🧩 **实体交互（Tangible）** | 物理探针方块放置在实体网格垫上，触发数字孪生端的数据响应，消除纯软件的隔离感 |
| ⚔️ **对抗驱动（PvP）** | 甲方设障 vs 乙方排故的不对称对抗，创造持续紧张的博弈张力 |
| 🔬 **因果逻辑（Causal）** | 所有故障组合须通过"逻辑规则引擎"验证其物理可行性，杜绝伪故障 |
| 🎓 **职教根植（Vocational）** | 内容贴合国家职业技能标准（化工、制药、电气方向），可直接对接大赛赛制 |

## 二、目标用户与痛点分析

### 2.1 目标用户画像

#### 主要购买方：职业院校（中职 / 高职）

- **学校类型**：化工类、制药类、食品类、电气自动化类中高职院校
- **决策链**：实训中心主任 → 系部主任 → 分管教学副校长
- **关键驱动**：职业技能大赛备赛、实训设备采购更新、"1+X"证书配套教学

#### 次要用户：学生

- **中职学生（15–18岁）**：动手欲强，短视频游戏原住民，对枯燥的工艺图深感厌倦
- **高职学生（18–21岁）**：面临顶岗实习和就业压力，渴望真实的复杂系统处置能力

#### 延伸用户：企业培训

- 化工、制药企业新员工上岗前安全培训，以及资深操作员的年度应急演练

### 2.2 核心痛点矩阵

| 痛点维度 | 现状描述 | 严重程度 |
|----------|----------|----------|
| **趣味性缺失** | 传统仿真软件为单机点击模式，反馈仅为"操作正确/错误"红字提示，学生普遍形容为"高科技点读机" | ⭐⭐⭐⭐⭐ |
| **场景可重玩性低** | 每套实体模型（管路/装置）造价 5–50 万元，场景固定，毕业后即闲置；软件剧本有限，用过即弃 | ⭐⭐⭐⭐⭐ |
| **认知层次浅** | 现有软件以"已知故障位置→操作修复"为主，缺乏"未知故障→系统推理→定位→修复"的高阶训练 | ⭐⭐⭐⭐⭐ |
| **沟通协作缺训练** | 工厂事故处置的核心是多岗位协作（TRM），而实训绝大多数为个人操作，与真实岗位脱节 | ⭐⭐⭐⭐ |
| **硬件昂贵且静态** | 实体装置无法更换工艺，一套只能对应一个产品方向，跨专业通用性极差 | ⭐⭐⭐⭐ |
| **教学评估粗糙** | 绝大多数系统只能给出"完成/未完成"的二元评分，无法诊断学生的推理路径和思维盲区 | ⭐⭐⭐ |

## 三、核心玩法与机制——"甲乙方"UGC对抗引擎

### 3.1 游戏大循环（一局对抗流程）

```
【甲方：故障设计师】         【乙方：现场侦探组】
        │                           │
  ①领取"成本预算"             ①大屏显示工艺流程图
        │                      （含误导性异常症状）
  ②从"因果逻辑模块库"                │
    中选择并组合故障卡牌         ②讨论推理，消耗
        │                      "行动点"在物理沙
  ③系统验证逻辑合法性            盘网格上放置
        │                      "探针/工具方块"
  ④合法组合提交，系统                │
    注入虚拟工厂                ③获取该节点真实
        │                      数据（或发现是
  ⑤等待乙方探查，旁观                 正常节点）
    对手"被耍"的全过程               │
        │                      ④缩小嫌疑范围，
  ⑥乙方找到根因或行动               找到根本原因
    点耗尽后，结算得分                │
        └───────── 结算复盘 ──────────┘
```

### 3.2 甲方机制——"故障设计师"（The Trapper）

#### 3.2.1 成本预算系统（Cost Budget）

- 每局开始，甲方获得固定"成本预算点数"（例如：标准局 = 12点）
- 成本点数代表在真实工厂中"一次事故涉及的异常节点数量与稀有度"
- 简单故障（如单阀门卡死）消耗少量点数；复杂故障（如换热器内漏叠加仪表漂移）消耗大量点数

#### 3.2.2 因果逻辑模块库（Causal Logic Module Cards）

每张"故障卡"描述一个**物理上真实可信的设备异常状态**，包含：

| 字段 | 说明 |
|------|------|
| **故障名称** | 例：`P-101A 泵体机封泄漏` |
| **物理类别** | 机械 / 仪表 / 工艺 / 公用工程 |
| **上游前提条件** | 触发该故障需要什么先决条件（可为空） |
| **下游传播效应** | 该故障会导致哪些下游参数变化（含"误导性症状"标记） |
| **隐蔽等级** | 1–5星，越高则越难通过仪表数据直接定位 |
| **成本点数** | 1–5点 |

> ⚙️ **逻辑规则引擎**会在甲方提交前自动检验：所选故障卡的组合是否存在物理矛盾（例如同一条管线不能同时"全堵"和"全漏"）。**不合法的组合将被拒绝**，防止甲方创造无解局面。

#### 3.2.3 迷惑性症状生成

系统基于故障组合，自动在DCS大屏上**按优先级注入误导性警报**：

- 真实根因导致的直接症状（低权重显示，或被噪声掩盖）
- 由传播效应引发的下游次生症状（高权重显示，最先报警）
- 部分与故障无关的随机"背景噪声"警报（可由甲方消耗额外点数购买）

### 3.3 乙方机制——"现场侦探组"（The Detective）

#### 3.3.1 行动点系统（Action Points）

- 每局乙方获得固定"行动点"（AP，例：标准局 = 20AP）
- 将探针/工具方块放置到沙盘网格的某个节点上，消耗AP（依节点重要性和距离计算）
- AP耗尽前未找到并修复根因，则判定乙方失败

#### 3.3.2 物理沙盘网格探查

乙方在实体沙盘的**网格坐标**上放置不同功能的**实体方块**：

| 方块类型 | 功能 | AP消耗 |
|----------|------|--------|
| 🔵 **压力表探针** | 读取该节点当前压力 | 1 AP |
| 🟡 **温度计探针** | 读取该节点当前温度 | 1 AP |
| 🔴 **超声波检测仪** | 判断该管段是否存在内漏 | 3 AP |
| 🟠 **红外热像仪** | 扫描周围3格节点的热分布 | 4 AP |
| 🔧 **隔离阀操作块** | 将某节点切出系统（验证假设） | 2 AP |
| 🔩 **维修操作块** | 对已确认故障节点执行修复 | 3 AP |

放置后，该节点的**真实数据**显示在DCS大屏的对应位置，并在复盘记录中留存时间戳。

#### 3.3.3 多角色协作模式（TRM 团队资源管理）

在多人对局中（推荐3–5人/队），乙方成员被分配不同**岗位权限**：

| 岗位 | 权限 | 限制 |
|------|------|------|
| **主控室操作员** | 唯一可以操作DCS大屏、调整工艺参数的角色 | 不能触碰实体沙盘 |
| **现场巡检员** | 唯一可以在沙盘上放置探针方块的角色 | 看不到大屏全图，只能通过语言获取信息 |
| **维修工程师** | 唯一可以放置"维修操作块"的角色 | 必须先收到现场巡检员的"故障确认"指令才可操作 |
| **班长/指挥** | 可观察全局，但没有直接操作权限，负责协调沟通 | — |

> 💡 **设计意图**：岗位权限的信息不对称，迫使成员之间进行精准、高效的口头沟通——这正是工业TRM训练的核心。系统后台全程录制对话和操作时间轴，供课后复盘。

### 4.2 硬件关键设计决策

#### 4.2.1 智能网格垫——两种方案对比

| 方案 | 技术路线 | 单套硬件成本估算 | 优势 | 劣势 |
|------|----------|------------------|------|------|
| **方案A（推荐MVP）** | 顶置摄像头 + OpenCV视觉识别方块上的ArUco码 | 约 ¥800–1200 | 成本最低；无需改造垫子；可实时录像 | 受光线影响；方块被遮挡时失效 |
| **方案B（量产优选）** | 柔性PCB NFC天线矩阵嵌入垫内 + 方块底部NFC标签 | 约 ¥3000–5000 | 稳定可靠；不受光线影响；响应更快 | 前期开发成本较高 |

#### 4.2.2 可更换工艺底图纸

- 材质：防水哑光PVC覆膜，A0尺寸，印刷工艺P&ID简化图
- 包含：节点坐标网格（字母+数字双轴标注）、关键设备图例、安全色彩编码
- 扩展方式：不同专业方向（化工合成/制药提取/石油炼化）购买对应底图包，即可在同一硬件平台切换
- 单张底图印刷成本：约 ¥15–30

#### 4.2.3 通用功能方块

- 外形：4cm × 4cm × 4cm 正方体，顶部嵌入1.54寸E-Ink小屏（显示方块类型图标）
- 材质：ABS工程塑料，工业级耐摔设计
- 识别标识：底部NFC标签（方案B）或四面ArUco码贴纸（方案A）
- 一套标准配置：共计30个方块（含各类探针10个、工具块10个、操作块10个）

#### 4.3.1 逻辑规则引擎（Logic Rule Engine）——核心竞争壁垒

该引擎是ILA产品的技术护城河，主要功能：

1. **因果图谱管理**：以有向图（DAG）表示化工流程中设备故障的传播路径，每条边附有物理条件和传播延迟
2. **甲方合法性验证**：提交前实时检测故障组合是否存在拓扑矛盾，确保每一局都有且仅有一个逻辑自洽的根因
3. **症状权重演算**：根据故障组合在因果图上的传播深度，自动分配各症状的"显眼度"和"迷惑性系数"，生成DCS大屏的告警序列
4. **可扩展卡牌API**：内容合作伙伴（高校教师、行业专家）可通过标准化JSON Schema提交新的因果逻辑模块，经平台审核后发布

## 五、教育价值主张

### 5.1 布鲁姆教育目标层次对应

| 布鲁姆层次 | 传统仿真软件 | 工业逻辑竞技场 |
|-----------|------------|--------------|
| **记忆** | ✅ 工艺流程图记忆 | ✅ 卡牌内容学习 |
| **理解** | ✅ 单故障原理解释 | ✅ 因果逻辑关系理解 |
| **应用** | ✅ 标准操作规程执行 | ✅ 探针选择与部署 |
| **分析** | ❌ 极少 | ✅ **复杂症状反向推理根因** |
| **评估** | ❌ 无 | ✅ **甲方故障设计的经济性与迷惑性评估** |
| **创造** | ❌ 无 | ✅ **UGC故障剧本设计（高阶模式）** |

> 💡 ILA将训练重心从"记忆/理解"层次**向上拉升至分析、评估、创造层次**，这是传统仿真软件无法企及的核心价值差异。

### 5.3 可量化学习成果

- **故障根因定位准确率**：追踪学生的探查路径效率，识别高频思维盲区
- **行动点利用率**：衡量学生的资源管理与推理效率
- **团队沟通质量指数**（多人模式）：基于操作时间序列分析协作协同度
- **UGC故障设计合法率**（Level 3）：反映学生对因果逻辑的掌握深度
- 所有数据可导出**PDF诊断报告**，直接作为教评材料归档

## 六、商业化策略

### 6.1 商业模式——"买硬件，订内容"

```
收入来源结构
```

### 6.2 定价策略与竞争对比

| 对比维度 | 传统3D仿真软件 | 实体装置模型 | **工业逻辑竞技场** |
|---------|-------------|------------|-----------------|
| 初始采购成本 | ¥5–20万/套 | ¥10–100万/套 | **¥1.5–2.5万/套** |
| 年度维护/更新成本 | ¥1–3万/年 | ¥0.5–2万/年（维修） | **¥0.3–0.5万/年** |
| 场景可替换性 | 中（软件升级） | 低（实体固定） | **极高（换底图纸即切换）** |
| 多人对抗支持 | 无 | 无 | **原生支持** |
| UGC内容生态 | 无 | 无 | **核心竞争力** |
| 技能大赛适配 | 有限 | 有限 | **专项设计** |

### 6.3 渠道策略

- **直销渠道**：直接面向中高职院校实训主任，参加职教装备展（如每年中国国际职业教育博览会）
- **代理渠道**：与职教出版社、教育装备代理商合作，进入其现有销售网络
- **赛事绑定**：争取成为全国职业技能大赛（化工生产技术赛项）**指定设备合作伙伴**
- **示范校策略**：优先在3–5所有影响力的国家示范性高职布局，形成口碑标杆

### 6.4 竞争壁垒

1. **逻辑规则引擎的内容深度**：因果卡牌库的积累是长期壁垒，竞争对手难以快速复制
2. **UGC生态的网络效应**：教师投稿的剧本越多，平台越有价值，形成正向飞轮
3. **大赛赛制深度绑定**：一旦成为指定赛事设备，形成教学倒逼采购的强驱动力
4. **硬件成本护城河**：极简的硬件方案让竞品的"炫技"方案（AR眼镜、昂贵沉浸舱）在职教采购面前毫无优势

## 八、风险与应对

| 风险类型 | 风险描述 | 应对措施 |
|---------|---------|---------|
| **教学安全顾虑** | 保守型教师担心"游戏化"降低学生对高危行业的敬畏心 | 强调每张故障卡均对应真实工业事故案例，产品定位为"硬核推理工具"而非娱乐游戏；配套出版教学指南 |
| **硬件稳定性** | 摄像头视觉识别在强光/阴影下失效影响课堂 | MVP阶段同步测试NFC方案；给每个方块提供手动输入备用通道 |
| **采购预算限制** | 职校设备采购周期长，预算有限 | 提供"软件先行"订阅方案（学校用自己的桌面+手动录入暂代沙盘）；配合申报教育部产教融合专项资金 |
| **逻辑引擎内容不足** | 初期故障卡牌库小，可玩性受限 | 与合作院校共建内容；优先聚焦1–2个专业方向深耕，而非铺开 |
| **竞品快速模仿** | 大型仿真软件厂商快速推出类似功能 | 硬件+软件+UGC生态的组合壁垒难以快速复制；尽快完成赛事绑定建立先发优势 |

## 九、附录：多角色推演纪要摘要

> 本产品策划书的核心创意，源自一次由9名虚拟角色参与的**多角色对抗头脑风暴**推演。以下为关键转折点摘要：

### 9.1 关键否定时刻（推动创意升华的转折点）

**【否定1】中职老师B对"放方块看爆炸"玩法的否定**

> *"这简直是在教学生犯罪！化工制药容不得儿戏，随便扔故障方块看爆炸，会在学生潜意识里形成危险的娱乐化倾向。"*

→ **转化成果**：将"放方块制造故障"改为"放方块进行探查"——方块成为侦探工具而非破坏工具，实现了安全性与趣味性的统一。

**【否定2】高职老师D对"大屏亮红灯告诉学生哪里坏了"玩法的否定**

> *"这叫开卷考试，毫无意义。化工的核心难点不是'知道哪里漏了去堵'，而是'根本不知道哪里漏了，只能通过仪表数据倒推'。"*

→ **转化成果**：确立了"黑盒排查"为核心游戏机制——故障位置始终对乙方隐藏，大屏只显示间接症状，形成了产品最核心的差异化价值。

**【否定3】技术大牛对AR眼镜方案的自我否定**

> *"AR眼镜成本太高且容易眩晕，先砍掉。回归极简：顶置摄像头+打印底图，成本能砍掉70%。"*

→ **转化成果**：确立了"极简硬件+丰富软件内容"的商业模式，形成了与竞品"炫技路线"的鲜明差异。

### 9.2 关键灵感时刻

**【灵感1】高职老师C提出"方块是探针，而非故障"**

> *"小方块不应该代表'故障'，小方块应该代表'传感器'或'探伤仪'！沙盘是一个被黑盒化的复杂管网！"*

→ 这一灵感是整个产品概念的**核心转折点**，彻底重塑了硬件交互逻辑。

**【灵感2】销售提出多角色岗位分工（TRM模式）**

> *"学生A是主控室操作员只能看大屏；学生B是现场巡检员只能摸沙盘；他们必须疯狂沟通才能找到真正的故障。"*

→ 将产品从"个人排故工具"升华为"团队协作训练系统"，显著扩大了教育价值维度。
