## 一、产品概述

### 1.1 产品名称与定位

| 项目 | 内容 |
|------|------|
| **中文名称** | 工业逻辑竞技场 |
| **英文名称** | Industrial Logic Arena |
| **产品定位** | 全球首款"虚实协同 + UGC回合制对抗"工业排故推演沙盘 |
| **目标用户** | 职业院校（中职/高职）化工、制药、电力等工艺类专业学生及教师 |
| **核心技术** | Aspen 工业过程仿真引擎 + 游戏化 Cost/Action 机制 + 虚实协同沙盘 |

### 1.2 产品愿景

> 将枯燥的化工/制药参数分析，转化为一场硬核的"团队资源管理（TRM）与黑盒逻辑推理竞技赛"。  
> 让学生为了**战胜对手**，而疯狂地去钻研底层工艺规律——这是任何传统填鸭式仿真软件都做不到的奇迹。

## 二、市场痛点分析

### 2.1 职教四大核心痛点

#### 痛点一：传统仿真软件"点读机化"
- 操作照搬答案，缺乏空间感与动手参与感
- 学生上课极其枯燥，参与度低下
- 学会了"怎么拧螺丝"，缺少"黑盒推理"能力培养

#### 痛点二：物理沙盘"盆景化"
- 造价极其昂贵（数十至上百万元），买来只能看不能动
- 工艺方案一改，整套设备随之报废
- 投资回报率极低，沦为展示道具

#### 痛点三：单机剧本"消耗快"
- 官方开发几个固定剧本，学生玩过一次即知晓答案
- 复玩率极低，设备沦为吃灰的摆设
- 内容更新依赖厂家，周期长成本高

#### 痛点四：缺乏"非技术技能"训练
- 真实工厂最需要的是：极端压力下的**团队沟通**、**欺骗性仪表甄别**能力
- 现有软件全都只在教"怎么操作设备"，不教"如何在混乱中保持逻辑清晰"

## 三、核心产品机制

### 3.1 底层引擎：Aspen 真实仿真驱动

不同于传统的"预设剧本式"教具，工业逻辑竞技场以 **Aspen Plus / Aspen HYSYS** 等工业级工艺仿真软件为物理引擎。

**核心优势：**
- 所有"干预（捣乱）"操作均直接作用于真实仿真模型，Aspen 自动演算全套连锁响应
- 故障后果百分之百符合热力学、流体力学与化学反应规律，杜绝"伪科学"
- 每一次对抗均可能产生独一无二的复杂并发故障场景，实现真正意义上的"无限复玩"

**分级权限约束（防止极端破坏）：**

| 难度等级 | 允许的干预范围 | 示例 |
|---------|--------------|------|
| 新手 (Lv.1) | 单点设备异常 | 单个阀门开度调整 |
| 进阶 (Lv.2) | 双点协同异常 | 冷却系统泄漏 + 进料流量变化 |
| 专家 (Lv.3) | 多变量复合扰动 | 催化剂失活 + 温控传感器偏移 + 物料杂质超标 |
| 大师 (Lv.4) | 跨系统级联故障 | 完全模拟真实工厂突发事故场景 |

### 3.2 游戏化核心机制：Cost / Action 双资源体系

#### 甲方（造局者/出题方）资源池

| 资源类型 | 说明 |
|---------|------|
| **破坏预算（Sabotage Points, SP）** | 每局固定配额（如 10 SP），限制甲方总破坏烈度 |
| **行动点（Action Points, AP）** | 每回合有限行动次数，推动甲方精打细算如何布置扰动 |
| **伪装币（Disguise Tokens）** | 可购买"虚假报警"、"传感器偏移"等干扰手段，消耗额外 SP |

**SP 成本示例：**

| 干预操作 | SP 消耗 |
|---------|---------|
| 单个阀门轻度内漏 | 2 SP |
| 冷却水管局部堵塞 | 3 SP |
| 催化剂活性缓慢下降 | 4 SP |
| 温度传感器读数偏移（传感器故障伪装） | 5 SP |
| 进料流量计量误差叠加（最隐蔽） | 6 SP |

#### 乙方（破局者/排查方）资源池

| 资源类型 | 说明 |
|---------|------|
| **探测预算（Investigation Points, IP）** | 每次在沙盘上放置探针读取数据均消耗 IP |
| **维修点（Repair Points, RP）** | 锁定根源后实施修复消耗，修错则同样扣除 |
| **分析时限** | 每回合有倒计时压力，超时双方平局 |

#### 胜负判定

```
乙方完全定位所有故障根源 AND 剩余资源 > 0  →  乙方（破局者）胜利
乙方资源耗尽 OR 误诊超过阈值  →  甲方（造局者）胜利
```

### 3.3 回合制 PVP 对抗流程

```
┌─────────────────────────────────────────────────────────┐
│                    一局对抗流程                           │
├──────────┬──────────────────────────────────────────────┤
│ 准备阶段 │ 双方确认难度等级，系统初始化 Aspen 仿真模型     │
├──────────┼──────────────────────────────────────────────┤
│ 布局阶段 │ 甲方（出题组）在 SP 预算内选择干预操作，        │
│          │ Aspen 自动演算并锁定故障状态（乙方不可见）      │
├──────────┼──────────────────────────────────────────────┤
│ 大屏呈现 │ 系统将故障叠加后的"复杂表象参数"呈现在大屏：    │
│          │ 多点报警、异常数值、混淆信号同时出现            │
├──────────┼──────────────────────────────────────────────┤
│ 排查阶段 │ 乙方（解题组）在沙盘上放置探针方块（每次消耗    │
│          │ IP），解锁该节点真实参数，逐步剥离伪装          │
├──────────┼──────────────────────────────────────────────┤
│ 锁定阶段 │ 乙方锁定所有故障根源，放置"维修方块"修复        │
├──────────┼──────────────────────────────────────────────┤
│ 结算阶段 │ Aspen 验证修复操作，系统公布胜负及完整因果链复盘 │
└──────────┴──────────────────────────────────────────────┘
```

### 3.4 特殊能力卡/道具系统

| 卡牌名称 | 类型 | 效果 | SP/IP 消耗 |
|---------|------|------|-----------|
| 黑盒加密 | 甲方攻击卡 | 隐藏某节点的真实参数 2 轮 | 3 SP |
| 虚警喷射 | 甲方攻击卡 | 在 3 个随机节点同时触发虚假报警 | 4 SP |
| 全局扫描 | 乙方防御卡 | 低精度扫描全系统，标记异常区域 | 3 IP |
| 专家会诊 | 乙方工具卡 | 精确解锁某节点全参数（高精度）| 5 IP |
| 时间加速 | 通用卡 | 延长本方回合时限 60 秒 | 2 SP/IP |
| 因果复现 | 教学辅助卡 | 回放某一故障的完整传导路径 | 免费（教学模式）|

## 四、虚实协同沙盘硬件设计

### 4.1 极简硬件理念（成本压缩至最低）

**放弃传统昂贵3D实体模型，采用：**

```
┌─────────────────────────────────┐
│    智能感应桌垫（NFC矩阵/视觉定位）  │  ← 可更换工艺底图纸（化工/制药/电力）
├─────────────────────────────────┤
│    通用探针方块（5-8个）           │  ← 代表不同检测手段（红外/超声波/压力表）
├─────────────────────────────────┤
│    中央大屏（DCS 仿真界面）        │  ← 呈现 Aspen 仿真参数与报警状态
└─────────────────────────────────┘
```

**优势：**
- 一套桌子适配化工、制药、电力、石化多专业（只需更换底图）
- 硬件成本大幅降低（相较传统沙盘降低 90%+）
- 设备便携，可在普通教室部署，无需专用实训室

### 4.2 硬件-软件交互协议

- 探针方块通过 **NFC / RFID** 与桌垫通信，定位坐标发送至后端
- 后端将坐标映射到 Aspen 仿真模型中的对应工艺节点
- Aspen 返回该节点当前真实参数（或伪装参数，由甲方布局决定）
- 参数结果实时更新至大屏 DCS 界面

## 五、教学价值体系

### 5.1 学习动机转变

```
传统教学：  要我学  →  枯燥完成任务
工业逻辑竞技场：  我要赢  →  主动研究工艺原理 + 故障特征
```

**费曼学习法最高境界的实现：**  
为了给对面设置最有迷惑性的故障（甲方），学生必须深度研究"哪些参数异常组合最难被识别"。  
为了在混乱参数中找到真正根源（乙方），学生必须真正理解工艺系统的因果逻辑。

### 5.2 强制 TRM（团队资源管理）训练

| 角色 | 职责 | 对应真实岗位 |
|------|------|------------|
| 中控员（1人） | 监视大屏 DCS 参数，指挥排查方向 | 中央控制室操作员 |
| 巡检员（2人） | 在沙盘上放置探针，获取现场数据 | 现场巡检工程师 |
| 决策员（1人） | 综合信息做最终锁定判断 | 班组长/工程师 |

**视角强制分离**：中控员看屏幕，巡检员看沙盘，双方必须语言沟通才能联合破局。

### 5.3 双轨激励体系

| 成就体系 | 内容 | 目的 |
|---------|------|------|
| **技术成就** | 正确识别故障类型、掌握特定工艺原理 | 对齐教学目标，保证知识习得 |
| **竞技排名** | 排行榜、胜率统计、最快破局记录 | 维持游戏驱动力，提高复玩率 |

## 六、软件架构

### 6.1 技术栈

| 层次 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | Vue.js 3 + Pinia | 大屏 DCS 界面、学生操控界面 |
| 后端 | Node.js + Express | 游戏逻辑引擎、会话管理 |
| 仿真引擎 | Aspen Plus / HYSYS API | 工艺过程实时演算 |
| 数据库 | MySQL | 用户数据、对战记录、成就系统 |
| 消息通信 | EMQX (MQTT) | 沙盘硬件与服务端实时通信 |
| 容器化 | Docker + Docker Compose | 标准化部署 |

### 6.2 "逻辑链叠加渲染器"核心设计

不需要开发实时物理仿真引擎，核心架构为：

```
甲方干预操作
    ↓
Aspen API 调用（设置扰动参数）
    ↓
Aspen 自动演算（物理引擎）
    ↓
获取故障状态下全系统参数快照
    ↓
叠加"伪装层"（甲方购买的欺骗道具）
    ↓
渲染至大屏 DCS 界面（乙方可见部分）
    ↓
乙方探针触碰节点 → 后端解密该点真实参数 → 实时推送
```

## 七、商业模式

### 7.1 产品分层销售

| 产品包 | 内容 | 目标客户 |
|-------|------|---------|
| **基础竞技包** | 沙盘硬件 + 软件平台（化工专业） | 中职院校 |
| **高级竞技包** | 基础包 + 多专业底图 + 高阶难度解锁 | 高职院校 |

- **内容更新订阅**：定期发布新专业底图、新难度卡牌、新故障场景包
- **赛事举办服务**：为院校提供"全国职教杯·工业逻辑竞技大赛"赛务系统
- **数据分析服务**：为教师提供班级学习分析报告（哪些故障类型最难，哪些学生需要补强）

## 八、竞争壁垒

1. **Aspen 仿真深度集成**：与工业级仿真软件的 API 对接，非轻易可复制的技术护城河
2. **UGC 题库网络效应**：学生/教师持续贡献高质量对战记录，题库自增长
3. **赛事品牌效应**：一旦建立"全国性竞赛"IP，形成学校采购驱动力
4. **极简硬件壁垒**：通用硬件底座 + 专用软件，竞争对手难以同时做到低价与高真实度

## 十、附录：头脑风暴历程摘要

本产品策划来源于多轮深度头脑风暴，核心演进路径如下：

```
第一阶段：物理沙盘 + 固定剧本
    → 问题：剧本消耗快，复玩率低

第二阶段：引入 UGC 卡牌拼装出题机制
    → 问题：逻辑链条预设需大量内容创作，且难以保证科学严谨性

第三阶段：以 Aspen 工艺仿真引擎替代人工逻辑链
    → 突破：科学严谨性由 Aspen 自动保证，出题不再需要人工审核
    → 新问题：纯仿真缺乏竞技游戏性，学生失去参与感

第四阶段（当前版本）：Aspen 仿真 × Cost/Action 游戏机制深度融合
    → Aspen 做"物理裁判"，游戏化机制提供竞技深度与策略乐趣
    → 最终形态：科学性、竞技性、创新性三位一体
```
