## 一、产品定位与核心理念

### 1.1 背景定位

本产品为**"工业逻辑竞技场"**，是一套面向化工/制药/能源等高危行业职业教育的**虚实协同推演沙盘**。产品在前期《硬核工业侦探沙盘》的基础上，引入完整的竞技游戏化机制，形成**"工业真实性 × 竞技对抗性 × 回合策略性"**三位一体的复合体系。

### 1.2 核心设计理念

```
┌─────────────────────────────────────────────────────────────────┐
│                    工业 + 竞技 + 平衡 复合体系                    │
├─────────────────┬───────────────────┬───────────────────────────┤
│  物理真实层      │   策略资源层        │    竞技对抗层              │
│  (Aspen 引擎)   │  (Cost + Action)   │  (PVP + 卡牌 + 积分)      │
│                 │                   │                           │
│ • 真实工艺流程  │ • 预算约束         │ • 回合制甲乙对抗           │
│ • 真实故障数据  │ • 行动点消耗       │ • PVP 排名积分             │
│ • 参数联动关系  │ • 资源分配决策     │ • Bonus/Fine 奖惩          │
│ • 过程仿真反馈  │ • 风险回报权衡     │ • 特殊技能卡               │
└─────────────────┴───────────────────┴───────────────────────────┘
```

### 1.3 用户角色

| 角色 | 身份 | 核心诉求 |
|------|------|----------|
| **甲方（出题方）** | 高阶学生 / 教师 | 布置隐蔽故障，在有限预算内最大化迷惑性 |
| **乙方（解题方）** | 实训学生团队（2-4人）| 以最少代价在时限内定位并修复故障 |
| **裁判系统** | Aspen + 规则引擎 | 验证工程逻辑合规性，输出真实仿真数据 |

## 二、核心游戏机制设计

### 2.1 Aspen 物理引擎集成层

> **定位**：所有游戏元素的物理合法性由 Aspen（或同等过程仿真引擎）背书，确保教学科学性。

#### 2.1.1 引擎角色分工

```
甲方出题 ──→ 故障注入请求 ──→ [Aspen 合法性验证]
                                    │
                    ┌───────────────┼───────────────┐
                    ↓               ↓               ↓
               通过验证          验证失败          参数异常
               生成剧本        输出错误提示      触发警告卡
                    │
                    ↓
       Aspen 实时计算影响链（压力/温度/流量/组分变化）
                    │
                    ↓
       大屏 DCS 仿真界面展示"症状数据"（隐藏根因）
```

#### 2.1.2 仿真数据与游戏数据的映射

| Aspen 仿真输出 | 游戏化映射 |
|---------------|-----------|
| 反应釜温度偏高 3°C | 乙方收到"异常温度警报卡（黄色）" |
| 管线压降增大 15% | 乙方获得"压力异常线索" |
| 产品纯度下降至 92% | 大屏显示"产品质量扣分警报" |
| 流量计读数波动 | 触发"仪表干扰"特殊事件卡 |

#### 2.1.3 Aspen 对 Cost/Action 的反馈

Aspen 仿真结果直接影响资源管理：
- **故障恶化**：每延误 1 回合未修复，自动增加 10 Cost 损失（模拟停产损失）
- **操作失误**：乙方错误干预后 Aspen 返回"次生故障"，追加 Action 消耗
- **精准定位奖励**：第一次探测即命中故障节点，Aspen 触发"精准诊断"Bonus

### 2.2 Cost 预算能量机制

> **核心理念**：预算即约束，预算即压力。无限资源等于无教学价值。

#### 2.2.1 双方 Cost 池设计

```
┌─────────────────────────────────────────────────┐
│               甲方（出题方）Cost 池               │
├─────────────────────────────────────────────────┤
│ 初始布阵预算：20 CP（Cost Points）               │
│                                                  │
│ 故障卡消耗示例：                                  │
│  • 隐蔽微漏（Level 1）        ：  3 CP           │
│  • 仪表失灵（Level 2）        ：  5 CP           │
│  • 换热器结垢（Level 3）       ：  7 CP           │
│  • 多点联动故障（Level 4）     ： 12 CP           │
│                                                  │
│ 迷惑卡消耗示例：                                  │
│  • 布置虚假报警               ：  2 CP           │
│  • 干扰仪表（噪声注入）        ：  3 CP           │
│  • 设置"陷阱节点"             ：  4 CP           │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│               乙方（解题方）Cost 池               │
├─────────────────────────────────────────────────┤
│ 初始维修预算：30 CP                              │
│                                                  │
│ 探测工具消耗示例：                                │
│  • 红外测温枪（单点）          ：  1 CP           │
│  • 超声波探伤仪（单点）        ：  2 CP           │
│  • 压力计接入（单线段）        ：  2 CP           │
│  • 全线路扫描（计算资源密集）   ：  8 CP           │
│                                                  │
│ 维修操作消耗示例：                                │
│  • 阀门更换                   ：  5 CP           │
│  • 传感器校准                 ：  3 CP           │
│  • 管线补漏                   ：  6 CP           │
│  • 错误维修（Aspen 验证失败）  ： +5 CP 惩罚     │
└─────────────────────────────────────────────────┘
```

#### 2.2.2 Cost 动态变化规则

```
Cost 每回合变化 = 基础消耗 + 故障恶化损耗 - 回合奖励

• 基础消耗：乙方每回合固定消耗 2 CP（模拟停产时间成本）
• 故障恶化：每个未修复故障每回合额外增加 1~5 CP 损失（按故障级别）
• 回合奖励：乙方正确锁定故障可获得 3 CP 返还
• Bonus 卡  ：特定事件触发临时 CP 补充（见第 2.6 节）
```

#### 2.2.3 Cost 可视化（大屏展示）

- **实时预算条**：甲乙双方各有可见的彩色预算条（红/绿）
- **消耗历史图**：每次操作后动态更新消耗折线图
- **危机预警**：乙方 CP < 10 时触发"资金危机"警报动画

### 2.3 Action 行动点机制

> **核心理念**：每回合的行动点代表"时间窗口"和"人力资源"。强迫玩家做优先级决策。

#### 2.3.1 Action 点基础规则

| 角色 | 每回合 AP（Action Points） | 用途 |
|------|--------------------------|------|
| 甲方 | 3 AP | 布置故障、激活迷惑卡、触发特殊事件 |
| 乙方（每人） | 2 AP | 探测操作、维修操作、使用技能卡 |
| 乙方（团队4人）| 8 AP/回合 | 可自由分配，但不可累积到下回合 |

#### 2.3.2 Action 消耗对照表

**乙方探测 Action 消耗**

| 操作 | AP 消耗 | 获取信息 |
|------|---------|---------|
| 目视检查（单节点） | 0.5 AP | 是否有明显异常 |
| 简单仪表读数 | 0.5 AP | 该节点实时数值 |
| 精密探测（红外/超声）| 1 AP | 详细状态数据 |
| 多节点联合测试 | 2 AP | 区域关联数据分析 |
| 系统级诊断（DCS 联动）| 3 AP | 全局参数快照 |

**乙方维修 Action 消耗**

| 操作 | AP 消耗 | 需要 CP |
|------|---------|---------|
| 调整参数（软件操作）| 1 AP | 2 CP |
| 更换小型元件 | 1 AP | 3-5 CP |
| 更换大型设备 | 2 AP | 8-15 CP |
| 紧急停车操作 | 1 AP | 10 CP |

#### 2.3.3 AP 协作分配策略（乙方团队）

```
建议角色分工（4人团队）：
┌────────────────┬──────────────────────────────────────────┐
│  中控员（1人） │ 守大屏，分析 DCS 参数，消耗 AP 调用系统查询 │
│  巡检员（2人） │ 在沙盘上放置探测积木，消耗 AP 获取节点数据 │
│  协调员（1人） │ 规划 AP 分配、管理 CP 预算、打出技能卡      │
└────────────────┴──────────────────────────────────────────┘
```

#### 2.3.4 AP 透支规则

- **透支可行**：乙方可透支最多 2 AP（借用下一回合），但下回合 AP 减少 2
- **透支惩罚**：连续透支 2 回合，触发"超负荷"状态，该回合所有 AP 消耗翻倍
- **甲方 AP 保留**：甲方每回合未使用的 AP 可保留最多 1 个（代表"伏兵"储备）

### 2.4 回合制结构与甲乙方对抗规则

#### 2.4.1 对局完整结构

```
┌─────────────────────────────────────────────────────────────┐
│                       一局游戏流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ▶ 准备阶段（5 分钟）                                        │
│    • 甲方：在系统后台使用 CP+AP 布置故障（系统验证合法性）    │
│    • 乙方：分配角色，制定初步排查策略                        │
│                                                             │
│  ▶ 对抗阶段（最多 N 回合，N 由难度决定）                     │
│    ┌─────────────────────────────────────────────────────┐  │
│    │  每回合流程（限时 3~5 分钟）                          │  │
│    │  Step 1：甲方激活/追加干扰（使用 AP+CP）              │  │
│    │  Step 2：乙方获得新 AP，讨论并执行探测/维修操作        │  │
│    │  Step 3：Aspen 引擎结算本回合操作的真实影响            │  │
│    │  Step 4：大屏更新 DCS 数据（含甲方干扰后的参数）       │  │
│    │  Step 5：回合评分结算（Bonus/Fine 判定）               │  │
│    └─────────────────────────────────────────────────────┘  │
│                                                             │
│  ▶ 结算阶段                                                  │
│    • 乙方正确修复所有故障 → 乙方胜利                         │
│    • 乙方 CP 耗尽或超出回合上限 → 甲方胜利                   │
│    • 系统生成《诊断路径报告》，供教师评分                    │
│                                                             │
│  ▶ 攻守互换（可选）                                          │
│    • 完成一局后甲乙方互换，各打一局，总分合算                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 2.4.2 回合难度分级

| 难度 | 最大回合数 | 甲方初始 CP | 乙方初始 CP | 适用群体 |
|------|-----------|------------|------------|---------|
| 入门 | 5 | 10 CP | 40 CP | 中职一年级 |
| 进阶 | 8 | 15 CP | 30 CP | 中职二/三年级 |
| 硬核 | 12 | 20 CP | 25 CP | 高职 / 大赛选手 |
| 大师 | 15 | 28 CP | 20 CP | 院校间竞赛 |

#### 2.4.3 甲方出题规则约束

为防止甲方出不符合工程规律的"乱题"，建立三层约束：

**约束一：节点绑定约束**
```
故障卡 "阀门内漏"       ──→ 只能放置在"管道节点"
故障卡 "催化剂中毒"     ──→ 只能放置在"反应釜节点"
故障卡 "换热器结垢"     ──→ 只能放置在"换热器节点"
故障卡 "仪表零漂"       ──→ 只能放置在"仪表节点"
```

**约束二：因果绑定约束**
```
打出故障卡后，甲方必须从系统提供的"症状库"中
选择 1~3 个与该故障工程逻辑匹配的症状参数。

若选择的症状与故障不匹配，Aspen 验证失败 → 出题无效，
消耗的 CP 不予退还（惩罚随意出题）。
```

**约束三：预算上限约束**
```
单一故障的 CP 消耗上限 = 甲方总 CP 的 60%
（防止甲方用全部预算堆砌单一超强故障，保证游戏可玩性）
```

### 2.5 PVP 积分体系

#### 2.5.1 积分计算公式

**乙方得分**
```
乙方总分 = 基础分 + 速度加成 + 精准加成 + 团队加成 - 失误扣分

• 基础分       ：成功修复每个故障得 100 分
• 速度加成     ：每提前 1 回合结束，+20 分
• 精准加成     ：首次探测命中故障节点（无空探），每个 +30 分
• 团队加成     ：4人均有操作记录（避免1人包揽），+20 分
• 失误扣分     ：每次错误维修（Aspen 验证失败），-25 分
• CP 剩余加成  ：结束时剩余 CP × 2 分
```

**甲方得分**
```
甲方总分 = 迷惑加成 + 坚守加成 - 漏洞扣分

• 迷惑加成     ：乙方每次空探（探对了区域但找错了节点），+15 分
• 坚守加成     ：乙方超过一半回合才定位故障，+30 分
• 乙方超时/破产：+100 分（主要胜利分）
• 漏洞扣分     ：故障被"1步命中"定位，-30 分（倒逼甲方设计更隐蔽的题）
```

#### 2.5.2 赛季排行榜

```
赛季积分 = 本赛季所有对局（含甲方局+乙方局）的平均得分

排行榜分档：
★★★★★ 大师段位  (Top 5%)   ── 解锁"特级大师"专属故障卡
★★★★  专家段位  (Top 15%)  ── 解锁"专家版"题库
★★★   工程师段位 (Top 35%)  ── 解锁"中级"题库
★★    技工段位  (Top 65%)   ── 默认题库
★     学徒段位  (其余)       ── 简化题库
```

#### 2.5.3 联赛模式

```
校级联赛：班级间对抗，甲乙方互换各一局
区域联赛：院校间竞赛，引入裁判组（教师）担任甲方，增加难度
全国赛  ：官方指定标准剧本，统一评分，排行榜全国可见
```

### 2.6 Bonus / Fine 奖惩机制

> **核心理念**：奖惩机制是游戏节奏的调节器，也是教学行为正向引导的工具。

#### 2.6.1 Bonus（奖励）事件

| 触发条件 | 奖励内容 | 教学意图 |
|---------|---------|---------|
| 乙方正确描述故障前三条症状关联逻辑 | +15 分 + 2 CP 返还 | 强化逻辑推理表达能力 |
| 乙方仅使用"廉价探测工具"（无高价仪器）即定位故障 | +40 分 | 鼓励掌握基础技能 |
| 乙方团队无任何成员 AP 浪费（AP 利用率 >90%）| +20 分 | 强化团队协作效率 |
| 甲方设置的故障通过了 Aspen 最严格验证等级 | 甲方 +20 分 | 激励甲方深度钻研工艺 |
| 乙方通过"系统级诊断"一次锁定全部故障 | "精准排查"称号 + +50 分 | 奖励高端技能使用 |

#### 2.6.2 Fine（惩罚）事件

| 触发条件 | 惩罚内容 | 教学意图 |
|---------|---------|---------|
| 乙方错误维修导致 Aspen 返回次生故障 | -30 分 + 额外消耗 5 CP | 杜绝"乱修一气"的浮躁行为 |
| 乙方维修前未进行任何探测（盲修） | -20 分 | 强调诊断先行的工程规范 |
| 甲方布置的故障被 Aspen 两次验证拒绝 | 甲方 -10 分 + 减少 1 AP | 约束甲方不认真出题 |
| 乙方 CP 耗尽（破产出局）| 乙方 -50 分（本局失败） | 强化成本意识 |
| 操作超时（单步操作超过 90 秒未确认）| -5 分 / 次 | 培养决策效率 |

#### 2.6.3 连锁奖惩（连击系统）

```
连续正确探测（3次以上）→ 触发"黄金连击"模式：
  • 下一次探测 Cost 减半
  • 解锁额外"线索卡"（提示故障大类）

连续错误操作（2次以上）→ 触发"混沌状态"：
  • 本回合大屏参数显示噪声增加（模拟决策混乱）
  • 下一回合 AP 减少 1
```

### 2.7 特殊卡牌机制

> **核心理念**：特殊卡是游戏的"意外"和"翻盘"元素，也是对深度工业知识的奖励。

#### 2.7.1 卡牌获取途径

```
① 每局开始时，根据段位随机抽取 2~4 张手牌
② 完成特定教学任务（如通过理论考核）可解锁专属卡
③ 赛季积分达到阈值，自动获得限定卡
④ 正确回答系统随机弹出的"知识问答"即时获得随机卡
```

#### 2.7.2 卡牌分类与效果

**【甲方专属卡】**

| 卡名 | 消耗 | 效果 | 工业知识锚点 |
|------|------|------|------------|
| 🔴 连锁反应 | 2 AP | 选择一个已激活的故障，其影响扩散至相邻节点（Aspen 重新计算影响链）| 工艺联锁原理 |
| 🔴 雾霾干扰 | 1 AP + 3 CP | 本回合乙方大屏上所有温度数据显示误差 ±5% | 仪表精度与干扰 |
| 🔴 时间炸弹 | 3 AP | 设置一个"延时故障"，在 2 回合后自动激活 | 隐蔽型渐发性故障 |
| 🔴 镜像陷阱 | 2 AP + 2 CP | 在某节点布置虚假"症状"，乙方探测该节点会得到误导性正常数据 | 仪表虚值与串扰 |
| 🔴 双重故障 | 需已有 1 个故障 | 追加 1 个低级故障，且两者症状相互掩盖 | 多故障叠加诊断 |

**【乙方专属卡】**

| 卡名 | 消耗 | 效果 | 工业知识锚点 |
|------|------|------|------------|
| 🟢 专家会诊 | 2 AP | 系统提示当前最高可能性的故障大类（不具体到节点）| 系统性诊断思路 |
| 🟢 历史回溯 | 1 AP + 2 CP | 查看最近 2 回合的参数变化趋势（而非当前快照）| 趋势分析方法 |
| 🟢 应急停车 | 1 AP + 10 CP | 暂停系统恶化（本回合故障不再扩散），争取时间诊断 | 紧急停车 (ESD) 规程 |
| 🟢 知识图谱 | 0 AP + 0 CP | 显示当前工艺图上各节点的"关联关系图"（拓扑结构）| 过程系统工程 |
| 🟢 快速补丁 | 1 AP + 3 CP | 对已定位的故障进行"临时抑制"（不彻底修复，延缓恶化）| 临时处置规程 |

**【中立/双方均可用的卡】**

| 卡名 | 消耗 | 效果 |
|------|------|------|
| ⚡ 时间加速 | 3 CP | 跳过裁判系统的验证等待，立即获得结果 |
| 🔄 重置一项 | 2 AP | 撤回本回合的一次操作（AP 不返还）|
| 📊 数据广播 | 1 AP | 将某一节点的数据公开给所有人（增加透明度，适用于演示模式）|

#### 2.7.3 卡牌平衡约束

```
• 每局每张卡最多使用 1 次（避免无限叠加）
• 甲方手牌上限：4 张（防止堆砌策略）
• 乙方手牌上限：6 张（团队更多元化选择）
• 部分卡之间存在"互斥"关系（如"应急停车"和"快速补丁"不能同回合使用）
• "连锁反应"卡必须在 Aspen 验证通过后才能生效
```

## 三、平衡性与可扩展性设计

### 3.1 平衡性设计原则

#### 3.1.1 信息不对称平衡

```
甲方知道：故障位置、故障类型、已消耗 CP
乙方知道：大屏症状数据、已使用探测结果、剩余 CP/AP

核心张力："你知道我不知道什么" vs. "我会如何推断你知道的"
```

这种**不完全信息博弈**结构是游戏深度的来源，也对应真实工业排故场景（操作员不知道故障位置）。

#### 3.1.2 资源平衡设计

| 设计维度 | 平衡机制 |
|---------|---------|
| 甲强乙弱 | 乙方 CP 初始值总是高于甲方（代表企业维修资金充裕） |
| 乙强甲弱 | 甲方掌握故障位置信息优势，补偿资源劣势 |
| 新手保护 | 入门难度下，乙方获得"提示卡"×2，甲方禁用"雾霾干扰"卡 |
| 高手平衡 | 大师段位下，乙方 CP 与甲方相近，完全考验技术实力 |

#### 3.1.3 动态难度调整（DDA）

```
系统监控乙方连续失败次数：
• 连败 2 局 → 自动降低 1 个难度档
• 连胜 3 局 → 推荐升级难度档（非强制）
• 班级平均通关率 < 40% → 教师收到预警，推荐开展知识复习
```

### 3.2 可扩展性设计

#### 3.2.1 内容扩展（专业横向扩展）

```
基础版（化工）→ 扩展包 A（制药 GMP 生产线）
                → 扩展包 B（石油精炼/催化裂化）
                → 扩展包 C（新能源/电解水制氢）
                → 扩展包 D（污水处理工艺）
```

每个扩展包提供：
- 新的工艺底图（沙盘替换图纸）
- 专属故障卡库（基于 Aspen 该专业模型）
- 专属技能卡（专业知识锚点）
- 独立段位与排行榜

#### 3.2.2 机制扩展（纵向深化）

| 阶段 | 新增机制 |
|------|---------|
| V1.0 | 基础 Cost + Action + 回合制 + 卡牌（本文档） |
| V1.5 | 引入"市场价格波动"：维修材料 CP 每回合价格随机变化 |
| V2.0 | 引入"多厂联网"：跨校实时联机对战 |
| V2.5 | 引入"AI 甲方"：由 AI 自动生成高难度剧本 |
| V3.0 | 引入"碳排放成本"：过程能耗计入 CP，与绿色化工结合 |

#### 3.2.3 UGC（用户自制内容）扩展

```
学生出题工作台（简化版 Level Editor）：

Step 1：选择工艺底图（官方提供模板）
Step 2：在地图上点选节点，从卡牌库选择故障卡（系统自动进行 Aspen 验证）
Step 3：从症状库选择 1~3 个展示症状（不符合 Aspen 逻辑的症状标红警告）
Step 4：设置难度参数（CP 限制、回合数）
Step 5：提交审核（教师审核或 AI 自动审核）
Step 6：通过后发布到题库，创作者每次被选用获得"创作者积分"
```

## 四、引入机制的价值分析与潜在问题

### 4.1 价值提升分析

#### 4.1.1 教学科学性提升

| 引入机制 | 对应教学能力 | 提升幅度评估 |
|---------|------------|------------|
| Aspen 验证 | 工程逻辑严谨性 | ★★★★★ 根本性保障 |
| Cost 约束 | 工程经济意识、成本控制 | ★★★★☆ 显著提升 |
| AP 分配 | 项目管理、时间优先级判断 | ★★★★☆ 显著提升 |
| 回合制对抗 | 动态决策、压力下的逻辑推理 | ★★★★★ 核心竞争力 |
| 特殊卡牌 | 专业知识深度（获得卡须掌握知识）| ★★★☆☆ 辅助作用 |
| Bonus/Fine | 工程规范与操作纪律 | ★★★★☆ 行为正向引导 |

#### 4.1.2 产品商业价值提升

- **内容长尾**：UGC 机制使内容库自我增长，避免"玩完即弃"
- **联赛生态**：PVP 积分+排行榜天然形成竞技生态，增强口碑传播
- **复购模式**：专业扩展包提供持续营收，从"卖硬件"升级为"卖生态"
- **数据资产**：系统自动收集学生诊断路径数据，可为教研提供大数据分析

#### 4.1.3 差异化竞争优势

```
竞品对比：
传统仿真软件（如 Aspen Education）：有物理真实性，无游戏性，无对抗，无耐玩性
传统实训桌游（如卡牌排故游戏）  ：有游戏性，无物理真实性，教学深度有限
本产品"工业逻辑竞技场"          ：物理真实性 + 游戏竞技性 + 教学深度 = 唯一性
```

### 4.2 潜在问题与应对建议

#### 4.2.1 技术层面

| 潜在问题 | 风险等级 | 应对建议 |
|---------|---------|---------|
| Aspen 实时验证响应延迟影响游戏节奏 | 高 | 采用"异步验证"：操作立即执行，Aspen 在后台验证，若失败则下回合触发惩罚 |
| 多人同时操作沙盘导致并发冲突 | 中 | 引入"操作锁定"机制：同一节点同一时刻只允许一人操作 |
| Aspen 模型与沙盘工艺图不同步 | 中 | 建立"工艺图 - Aspen 模型"映射标准，使用版本管理 |

#### 4.2.2 教学层面

| 潜在问题 | 风险等级 | 应对建议 |
|---------|---------|---------|
| 学生过于专注"赢"而忽略工程学习 | 高 | 评分权重：学习过程（60%）> 胜负（40%）；并入课程总评成绩 |
| 甲方学生不认真出题（随便扣完 CP）| 中 | 引入"出题评分"：乙方解题后对题目质量打分，影响甲方积分 |
| 低难度模式下乙方 CP 远超需求，无压力 | 低 | 动态难度调整（DDA）自动收紧资源，教师可手动锁定资源配额 |
| 特殊卡导致局面失控，教学效果打折 | 中 | 教师可开启"无卡模式"，关闭所有特殊卡，回归纯逻辑推理模式 |

#### 4.2.3 产品运营层面

| 潜在问题 | 风险等级 | 应对建议 |
|---------|---------|---------|
| UGC 题库出现大量重复或低质量题目 | 中 | 设立"社区审核"机制：由高段位学生（专家段位+）担任审核员 |
| 赛季排行榜被单一院校垄断 | 低 | 分区赛制：院校内部排行榜与跨院校排行榜分开展示 |
| 新手因屡败放弃使用 | 高 | 专属"新手引导对局"：AI 甲方放水，确保前 3 局有成功体验 |

## 五、创新亮点汇总

### 5.1 核心创新点

1. **"Aspen + 卡牌"双重合法性机制**：每一张故障卡都必须通过 Aspen 物理引擎验证，让游戏策略建立在真实工程逻辑上，而非纯粹博弈。

2. **"信息不对称 + 资源约束"双重博弈层**：甲方拥有信息优势但资源有限；乙方拥有更多资源但面临信息黑盒。两种约束叠加形成高策略深度。

3. **"行动点 × 团队协作"强制机制**：AP 不可存储（除1点保留外），强迫乙方团队每回合都必须高强度协作，避免"一人包揽"的单机化体验。

4. **"出题即学习"的 UGC 飞轮**：甲方为了赢，必须深度研究工艺流程寻找刁钻故障；这本身就是最高效的主动学习方式。出题越多，系统题库越丰富，形成正向飞轮。

5. **"特殊卡即知识测验"**：特殊卡通过知识问答获得，将碎片化知识点转化为游戏资产，学得越多、牌越好、越强。

6. **"Aspen 过程数据 → 诊断报告"的教学评价闭环**：Aspen 自动记录每次操作的时间戳、效果和决策路径，生成可量化的《诊断能力评估报告》，替代主观评分。
